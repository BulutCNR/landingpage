Engagement scope

The methodology below reflects the full scope of a standard CNRCO engagement. Not every item applies to every system — scope is defined during the free scoping call and documented in a written agreement before work begins. Testing is performed only against systems the client owns and has authorized.

PHASE 01

Information Gathering and Reconnaissance

Before any active testing, a comprehensive picture of the target is built from publicly available and passively observable information. This establishes the attack surface and identifies assets the client may not have considered.

DNS and Domain Enumeration PASSIVE +
  • WHOIS record review — registrar, registration dates, contact exposure
  • DNS record enumeration — A, MX, TXT, CNAME, NS records
  • SPF record review — sender policy configuration and gaps
  • DKIM and DMARC configuration — email authentication hardening
  • Subdomain enumeration — finding forgotten or unprotected assets
  • Zone transfer attempt — misconfigured DNS exposing full zone data
HTTP Headers and Technology Stack PASSIVE +
  • Server header analysis — web server version disclosure
  • X-Powered-By and framework identification
  • Security headers audit — HSTS, CSP, X-Frame-Options, Referrer-Policy
  • Cookie attributes — Secure, HttpOnly, SameSite flags
  • CMS and library fingerprinting — WordPress, Drupal, known frameworks
  • JavaScript library versions — detection of outdated dependencies
SSL/TLS Certificate Inspection PASSIVE +
  • Certificate validity and expiry
  • Certificate chain completeness
  • Cipher suite strength — deprecated ciphers, weak key exchange
  • Protocol version — SSLv3, TLS 1.0, TLS 1.1 deprecation check
  • HSTS configuration and preload status
  • Certificate Transparency log review — unexpected issued certificates
Network Port and Service Discovery ACTIVE +
  • Open port scan — common and high port ranges
  • Service version detection on open ports
  • Unnecessary services — FTP, Telnet, unencrypted protocols
  • Admin panel exposure — phpMyAdmin, Adminer, default paths
  • Development or staging environment exposure
PHASE 02

Vulnerability Assessment

Automated scanning establishes baseline coverage. Manual testing is where context-aware assessment happens — identifying vulnerabilities that scanners miss because they require judgment about application logic, not just pattern matching.

SQL InjectionMANUAL+
  • Error-based injection — database version and structure disclosure
  • Blind boolean injection — inferring data without visible output
  • Time-based injection — detection via response delay
  • Union-based injection — extracting additional result sets
  • Second-order injection — payloads stored then executed later
  • ORM injection — parameterization gaps in abstraction layers

Tested on all user-controlled inputs interacting with the database: login forms, search fields, filters, URL parameters, API request bodies.

Cross-Site Scripting (XSS)MANUAL+
  • Reflected XSS — payload returned in immediate response
  • Stored XSS — payload persisted and served to other users
  • DOM-based XSS — client-side JavaScript processing user input
  • CSP bypass testing — evaluating Content Security Policy effectiveness
  • Filter bypass techniques — encoding, tag variation, event handler injection
Broken Access ControlMANUAL+
  • Horizontal privilege escalation — accessing other users' data by modifying IDs
  • Vertical privilege escalation — reaching admin functions from regular accounts
  • Insecure direct object reference (IDOR) — predictable resource identifiers
  • Missing function-level access control — API endpoints not checking authorization
  • Path traversal — accessing files outside the intended directory
  • Force browsing — accessing restricted pages without proper links

Access control testing requires understanding the application's permission model and attempting to circumvent it through direct manipulation, not just observing surface behavior.

Authentication and Session ManagementMANUAL+
  • Default credential testing
  • Brute force protection — rate limiting, lockout policies
  • Session token entropy — predictable or weak token generation
  • Session fixation — forcing a known session ID before authentication
  • Session invalidation — tokens remaining valid after logout
  • Password reset flow — token entropy, expiry, reuse prevention
  • Multi-factor authentication bypass attempts
API SecurityMANUAL+
  • API endpoint discovery — finding undocumented or legacy endpoints
  • Authentication requirement on all endpoints — including internal ones
  • Rate limiting on sensitive API operations
  • Input validation on all API parameters
  • CORS configuration — overly permissive origins
  • GraphQL introspection — information disclosure
  • Mass assignment — accepting unexpected parameters
  • Error message verbosity — stack traces in API responses
File Upload SecurityMANUAL+
  • Extension bypass — uploading executable files with allowed extensions
  • MIME type validation — content-type vs. actual file content
  • File size limit enforcement
  • Storage location — are uploaded files accessible from the web root?
  • Web shell upload attempts
Dependency and Component ScanningAUTO + MANUAL+
  • npm audit — known CVEs in Node.js dependencies
  • pip-audit — Python package vulnerability scan
  • Composer audit — PHP dependency review
  • WordPress plugin and theme version audit
  • Manual review of CVE severity in context of actual usage
Secrets and Configuration ExposureAUTO + MANUAL+
  • Hardcoded API keys in client-side JavaScript
  • .git directory exposure — source code and history accessible publicly
  • .env file exposure — environment variables served publicly
  • Backup file discovery — .bak, .old, .sql files in web root
  • Debug mode detection — detailed error output in production
  • Stack trace exposure in HTTP error responses
  • Directory listing — web server serving file indexes
PHASE 03

Risk Assessment and Prioritization

Every finding is scored using CVSS v3.1 and additionally ranked by business impact. CVSS provides a standardized technical severity; business impact adjusts for what the vulnerability actually means for your operations.

CVSS ScoreSeverityBusiness Impact DefinitionRemediation Priority
9.0 – 10.0CriticalBreach exposes customer data, payment info, or causes operational shutdownImmediate — 1 to 2 weeks
7.0 – 8.9HighCould lead to unauthorized access or significant data disclosureShort-term — within 1 month
4.0 – 6.9MediumExploitable but requires specific conditions or user interactionMedium-term — 1 to 3 months
0.1 – 3.9LowTheoretical risk or requires multiple chained exploitsLong-term — ongoing
0.0InfoUseful for further attacks but not immediately exploitableAwareness only

Vulnerability name and category

Description — what it is and why it matters
Proof of concept — how it was discovered and demonstrated during testing
Remediation — specific steps with code examples where applicable
Standards mapping — OWASP category, GDPR implications, PCI-DSS if applicable
03OWASP Top 10 Compliance Checklist — 2021 Edition
Status per category — compliant, non-compliant, partially compliant, not applicable
Finding references linked to detailed findings where violations exist
04Remediation Roadmap
Prioritized by severity and business impact
Effort estimate per item in hours
Complexity rating — determines sequencing with development resources
Step-by-step remediation guidance for each finding
05Audit Metadata and Sign-off
Audit date, duration, and methodology version
Auditor name — named accountability for all findings
Scope document — what was tested and what was excluded
Assumptions, limitations, and tools used

ROADMAP

Remediation Timeline

Immediate

1 to 2 weeks

Critical vulnerabilities

CVSS 9.0+ or direct exposure of sensitive data. These require attention regardless of development sprint cycles. Specific remediation steps with code examples are provided in the report.

Short-term

Within 1 month

High-severity issues

CVSS 7.0 to 8.9. Significant risk but typically requires more remediation effort — architectural changes, library upgrades, or refactoring authentication flows. Effort estimates guide sprint planning.

Medium-term

1 to 3 months

Medium-severity and architectural issues

Vulnerabilities requiring specific conditions to exploit, or larger architectural improvements. Security header hardening, logging improvements, access control refactoring.

Long-term

Ongoing

Low-severity and process improvements

Low-risk items, informational findings, monitoring setup, and security culture improvements. Dependency management processes, incident response procedures, audit trail implementation.
